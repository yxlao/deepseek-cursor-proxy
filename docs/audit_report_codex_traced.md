# DeepSeek Thinking Tool-Call Protocol Audit

Date: 2026-05-01

Scope:

- Source code under `src/deepseek_cursor_proxy/`
- Captured trace session: `trace-dumps/20260429T134418.628275Z-pid80943`
- DeepSeek protocol source: <https://api-docs.deepseek.com/guides/thinking_mode>

## Executive Summary

The captured trace mostly follows DeepSeek's thinking-mode tool-call protocol: every upstream assistant message that needed `reasoning_content` had it, every tool result matched a pending tool-call ID, all upstream requests used `thinking: {"type": "enabled"}` and `reasoning_effort: "high"`, and all 21 requests completed with HTTP 200.

However, the current code is not fully protocol-exact in all modes. I found three concrete compliance issues and two residual risks:

1. Current `transform.py` does not recognize the recovery notice string used in this dumped trace, so replaying these exact Cursor transcripts today would repeatedly drop recovered active history instead of continuing the tool-call chain.
2. `thinking="pass-through"` treats an omitted `thinking` field as non-thinking, but DeepSeek documents thinking mode as enabled by default.
3. Recovery notices are sent back upstream as assistant `content` on tool-call messages, even though that text was generated by the proxy, not DeepSeek.
4. Portable cache fallbacks can restore reasoning across changed system prompts or mode switches; this worked in the trace, but it is a heuristic rather than exact replay of a DeepSeek-generated message.
5. The tool-context detector treats a `system` message as a reasoning boundary, while the DeepSeek guide describes the boundary in terms of `user` messages.

## Protocol Requirements Used

From the DeepSeek Thinking Mode guide:

- Thinking mode defaults to enabled.
- OpenAI-compatible requests use `thinking: {"type": "enabled"}` or `{"type": "disabled"}`.
- Thinking effort supports `high` and `max`; `low` and `medium` map to `high`, `xhigh` maps to `max`.
- In thinking mode, `reasoning_content` is returned beside `content`.
- If no tool call happened between two `user` messages, old assistant `reasoning_content` is optional and ignored.
- If a tool call happened between two `user` messages, the assistant `reasoning_content` for that tool-call turn must be passed back in all subsequent requests.
- Correct tool-call handling appends the full assistant message, including `content`, `reasoning_content`, and `tool_calls`, then appends `tool` messages with matching `tool_call_id`s.

## Finding 1: Current Code Does Not Recognize This Trace's Recovery Boundary

Severity: High

Current code recognizes only:

- `[deepseek-cursor-proxy] Refreshed reasoning_content history.`
- `Note: recovered this DeepSeek chat because older tool-call reasoning was unavailable; continuing with recent context only.`

See `RECOVERY_NOTICE_TEXT`, `LEGACY_RECOVERY_NOTICE_TEXT`, and `has_recovery_notice` in `src/deepseek_cursor_proxy/transform.py`.

The dumped trace uses a different notice:

```text
[deepseek-cursor-proxy] Recovered this DeepSeek chat because older tool-call reasoning was unavailable; continuing with recent context only.
```

The trace summary code still recognizes this older prefix in `src/deepseek_cursor_proxy/trace.py`, but the request transformer no longer does. That means the current trace display layer and current protocol-repair layer disagree about what counts as a recovery boundary.

Evidence from replaying the dumped Cursor request bodies through current `prepare_upstream_request`:

| Request | Captured upstream msgs | Current replay upstream msgs | Current recovered | Current dropped | Current boundary |
| --- | ---: | ---: | ---: | ---: | --- |
| 13 | 3 | 3 | 9 | 38 | false |
| 14 | 5 | 3 | 9 | 40 | false |
| 15 | 7 | 3 | 9 | 42 | false |
| 16 | 9 | 3 | 9 | 44 | false |
| 17 | 11 | 3 | 9 | 46 | false |
| 18 | 13 | 3 | 9 | 48 | false |
| 19 | 15 | 3 | 9 | 50 | false |
| 20 | 17 | 3 | 9 | 52 | false |
| 21 | 19 | 3 | 9 | 54 | false |

Why this matters:

- The captured runtime correctly continued the active recovered chain after request 13.
- Current code would repeatedly recover from the latest user message for these same transcripts.
- This avoids the immediate 400, but it does not follow the protocol goal of preserving and passing back the active tool-call reasoning chain.

Recommendation:

- Add the trace's older `[deepseek-cursor-proxy] Recovered...` prefix to `LEGACY_RECOVERY_NOTICE_TEXT` handling in `transform.py`.
- Add a regression test that feeds request 14's style of transcript into current `prepare_upstream_request` and asserts `continued_recovery_boundary == True`.

## Finding 2: `pass-through` Misclassifies DeepSeek's Default Thinking Mode

Severity: High

DeepSeek documents thinking mode as enabled by default. Current code only repairs reasoning when the outgoing request explicitly has `thinking: {"type": "enabled"}`.

Relevant code:

- `config.thinking != "pass-through"` controls whether the proxy injects `thinking`.
- `thinking_enabled` is true only when `prepared["thinking"]` exists and equals `{"type": "enabled"}`.
- Recovery boundary handling and reasoning repair are gated by `thinking_enabled`.

Impact:

If config uses `thinking: pass-through` and the client omits `thinking`, the proxy forwards no `thinking` field. DeepSeek will still treat the request as thinking-mode by default, but the proxy will not patch missing `reasoning_content`. A Cursor-style tool-call history can then be forwarded upstream without required reasoning and trigger DeepSeek's 400 error.

The local audit script `scripts/audit_deepseek_protocol.py` also observes this behavior:

```text
OBSERVE pass-through mode treats an omitted thinking field as non-thinking; DeepSeek's documented default is thinking enabled.
```

Recommendation:

- Treat omitted `thinking` as enabled unless the request or config explicitly disables it.
- Keep `pass-through` from injecting a field, but do not let it disable reasoning repair.
- Add a test for `ProxyConfig(thinking="pass-through")` with omitted `thinking` and missing tool-call `reasoning_content`.

## Finding 3: Recovery Notices Are Replayed Upstream As Assistant Content

Severity: Medium

In recovery mode, the proxy emits a Cursor-visible notice by prefixing assistant `content`. Later, when Cursor sends that assistant message back, the proxy includes the notice in the upstream assistant message.

Captured request 14 shows an upstream assistant tool-call message with both proxy-generated recovery notice content and restored DeepSeek reasoning:

```json
{
  "role": "assistant",
  "content": "[deepseek-cursor-proxy] Recovered this DeepSeek chat because older tool-call reasoning was unavailable; continuing with recent context only.\n\n",
  "reasoning_content": "The user wants to extend a script to add three variables instead of two. Let me first read the current file to understand what we're working with.",
  "tool_calls": [...]
}
```

This is not exact protocol replay. DeepSeek's sample appends `response.choices[0].message`, but this `content` prefix was generated by the proxy, not DeepSeek.

Impact:

- The request can still pass DeepSeek validation.
- The model sees proxy bookkeeping text as if it were prior assistant content.
- Message signatures and cache lookup may need to rely on weaker keys, such as tool-call ID, because content no longer exactly matches the upstream response.

Recommendation:

- Strip proxy recovery notices from assistant `content` before sending messages upstream.
- Keep the notice Cursor-visible only, or express recovery state as a synthetic `system` message only in the reduced recovery request.
- Add tests for both current and legacy notice prefixes.

## Finding 4: Portable Cache Hits Are Useful But Not Strict Replay

Severity: Medium/Low

The proxy stores scoped keys and portable keys. Portable keys intentionally ignore some context, such as system prompt differences, so reasoning can be restored across Cursor Agent/Plan mode changes or model-family switches.

This worked in the trace:

- Requests 6-12 used portable fallback hits after mode/model surface changes.
- Request 19 switched from `deepseek-v4-pro` to `deepseek-v4-flash` and still had 6 portable hits.
- DeepSeek accepted those requests.

Protocol concern:

DeepSeek's documented flow is to pass back the `reasoning_content` generated in that tool-call turn. A portable hit can be correct when the assistant/tool-call message is genuinely the same message in a rewritten client transcript. It is less exact when the surrounding system prompt or upstream model changes, because the restored reasoning may not have been generated under the exact upstream request prefix now being sent.

Recommendation:

- Keep portable fallback because it solves real Cursor mode-switch problems.
- Mark portable hits as compatibility repairs in logs/trace.
- Prefer exact scoped hits whenever available, which the code already does.
- Consider making portable fallback configurable for strict debugging.

## Finding 5: `system` Messages Are Treated As Tool-Reasoning Boundaries

Severity: Low

`assistant_needs_reasoning_for_tool_context` returns false when it sees a prior `user` or `system` message while scanning backward.

DeepSeek's guide describes the requirement boundary as "between two `user` messages." It does not state that an inserted `system` message ends the tool-call turn.

Impact:

If a client inserts a `system` message inside a user turn after a tool result and before an assistant message, current detection may stop requiring `reasoning_content` too early. I did not find this pattern in the dumped trace.

Recommendation:

- Consider stopping only at `user` messages for the strict DeepSeek rule.
- If `system` should remain a boundary for local recovery design, document that assumption and add a test.

## Areas That Comply In The Captured Trace

The following protocol areas looked correct in the trace and current default-code path:

- All traced upstream requests explicitly sent `thinking: {"type": "enabled"}`.
- All traced upstream requests sent normalized `reasoning_effort: "high"`.
- No traced upstream request forwarded `temperature`, `top_p`, `presence_penalty`, or `frequency_penalty`.
- All assistant messages with `tool_calls` had string `reasoning_content`.
- All final assistant messages after tool results and before the next user boundary had string `reasoning_content`.
- Every `tool` message referenced a pending tool-call ID.
- Streaming response chunks preserved `reasoning_content` in the Cursor-facing stream while also mirroring it into visible `content`.
- Streamed tool-call reasoning is stored before `[DONE]` once tool-call IDs are known.

## Overall Verdict

For the captured enabled-thinking trace, the proxy did send valid DeepSeek tool-call requests and avoided missing-`reasoning_content` 400s.

For current code, the protocol is not exact in all cases. The most urgent fix is the recovery-boundary notice mismatch, because it means current code would mishandle the exact dumped traces after the first recovery. The next most important fix is treating omitted `thinking` as enabled for repair purposes in `pass-through` mode.
